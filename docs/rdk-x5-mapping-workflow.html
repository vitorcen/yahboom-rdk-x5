Mapping WorkflowGUI one-key mapping & map save

RDK X5 Robot · cartographer + Nav2 · 2026-07-14 · 已实施,板上切换/存图验证
TL;DR — 建图与导航互斥(都要发布 map→odom TF),用一条 systemd Conflicts= 关系做成一个开关:GUI 🗺 建图 开 = systemctl start mapping, systemd 自动把 nav2 换下场;手柄慢速遥控走一圈,浏览器地图画布实时看到 cartographer 的 /map 长出来;点 💾 存图 = map_saver_cli 覆盖 room.{yaml,pgm}(留一代 .bak)并自动切回导航。直接关开关 = 丢弃重来。 Mapping and navigation are mutually exclusive (both own the map→odom TF); one systemd Conflicts= edge turns the swap into a single GUI switch. Save overwrites the nav map (one .bak generation) and flips back to Nav2 automatically.

一、为什么是这个形态Why this shape

  • 互斥是物理事实,不是策略 — cartographer 与 AMCL 都发布 map→odom, 同时跑 = TF 打架、定位乱跳。与其在应用层判断"现在是什么模式",不如让 systemd 的 Conflicts=nav2.service 保证任何时刻最多一个在场:启建图自动停导航, 启导航自动停建图,零特殊情况。
  • 只启 cartographer 本体 — 厂商 map_cartographer_launch.py 会把雷达+ 底盘驱动再拉一份,和常驻的 ms200-lidar/nav-bringup 服务双开抢设备。 mapping.service 只跑 cartographer_launch.py (cartographer_node + occupancy_grid),传感器和底盘全部复用现有服务。
  • 不 enable,只 start — 建图是一次性操作,重启板子后绝不应该自动回到建图模式; nav2 保持 enabled,开机永远是导航形态。
  • GUI 画布零改动 — 浏览器本来就订阅 /map;建图时发布者从 map_server 换成 occupancy_grid(1 Hz),地图自然实时生长,一行前端代码都不用加。

三、操作流程Operating procedure

  1. GUI 仪表盘打开 🗺 建图 — nav2 自动停(约 20 s),cartographer 从车当前位置起图; 画布切到"地图"模式看 /map 实时生长。
  2. 手柄慢速遥控走一圈 — 0.2~0.3 速度、拐弯别甩,每个房间进去转一下,门口过道多走一次; 回到起点附近结束让回环闭合抹掉累积误差。单房间 5~10 min,全屋 15~30 min。
  3. 点 💾 存图 — map_save.sh:旧图备份 room.bak.* → map_saver_cli 覆盖 room.{yaml,pgm} → 停建图 → 起 nav2(新图生效, AMCL 自动种在地图原点 = 建图起点)。
  4. 建歪了想重来:直接关掉建图开关 = 丢弃本次,恢复导航(旧图不受影响);再开一次从头建。
⚠ 存图后车在哪 — nav2 重启会把 AMCL 种在地图原点(建图起点)。 按第 2 步"回到起点附近结束"操作,定位一开始就是对的;若结束在别处,需在浏览器里拖一次 初始位姿或开车走两步等 AMCL 收敛。 After save, AMCL is seeded at the map origin (= mapping start point) — end your mapping loop near where you started.

五、实战教训Lessons learned

  • 厂商建图 launch 不能直接用 — map_cartographer_launch.py 内嵌 laser_bringup(雷达+驱动),与常驻服务双开抢串口/设备;只取 cartographer_launch.py 这一半。
  • nav2 停止必然超时进 failed — 它的 launch 树 20 s 内停不完,SIGKILL 收尾, 单元状态留在 failed;若不 reset-failed 直接 systemctl start 可能被旧状态干扰。mapping_set(false) 与 map_save.sh 之后的 start nav2 前都已内置 reset-failed(与 nav_start.sh 的历史坑同源)。
  • 切换有 ~20 s 交接窗 — nav2 停止期间 AMCL 与 cartographer 可能短暂同时发 map→odom,画布上地图会跳一下;等建图开关稳定亮起再开始走圈即可。 TimeoutStopSec=20 已把窗口压到最短。
  • cartographer 配置沿用厂商 lua — max_range 3.5 m 偏保守 (MS200 标称 12 m),全屋建图要多贴墙走;先零改动跑通,嫌慢再调 cartographer_config.lua。

六、地图会"偏"吗 & 业界持续改图做法Map drift & lifelong mapping in industry

room.pgm 是静态文件,永不自己变。会漂的是 AMCL 定位:环境变了 (家具挪位)雷达和旧图对不上;或开机位置不在建图起点。小改动 AMCL 能容忍,大改动重建。

业界对"小物件持续变动"的分层处理(不是改静态图,而是分层叠加):

层次做法代表
秒级·临时障碍costmap 叠加层:静态图不动,雷达实时命中写入 obstacle/voxel layer,导航绕行,物体走了自动衰减。Nav2 默认已有,本车在用 Nav2 costmap_2d,STVL
天级·半永久变动lifelong mapping / 持续定位建图:定位的同时维护弹性 pose-graph,新扫描持续融合进图,定期序列化替换旧图。ROS2 生态的标准答案是 SLAM Toolbox 的 localization/lifelong 模式(可平替 AMCL) SLAM Toolbox,RTAB-Map 增量记忆
商用扫地机做法每次任务后台重跑局部建图,与主图做变化检测+图合并, 置信度够才提交;用户看到的图"自己慢慢变对"Roborock/iRobot 等私有实现

本车现状:第一层已覆盖(临时障碍能绕);第二层若以后想要"图自动跟着家里变", 路线是把 AMCL 换成 SLAM Toolbox localization 模式——留作候选实验,不急。
