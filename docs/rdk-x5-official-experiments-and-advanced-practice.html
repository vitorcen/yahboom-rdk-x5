三、推荐完成顺序

阶段 1：设备基线

备份镜像，盘点 topics、types、QoS、TF、频率和带宽。

退出条件：重启后可一条 launch 拉起底盘、雷达和相机。

阶段 2：控制基线

标定底盘，测直线、旋转、全向移动和里程计漂移。

退出条件：有误差表，不靠“看起来差不多”。

阶段 3：导航基线

比较 gmapping/cartographer，再接 Nav2、碰撞监控和速度平滑。

退出条件：固定任务重复 20 次可统计成功率。

阶段 4：边缘感知

相机检测或分割部署到 BPU，输出标准 ROS 2 消息。

退出条件：报告 FPS、P50/P95 延迟、温度和精度。

阶段 5：端云协同

Wi-Fi 发送压缩图像和 scan，4090 节点返回动作建议。

退出条件：断网、超时、旧动作都能在车端停止。

阶段 6：学习策略

先 LiDAR-to-Action，再图像融合，最后加语言条件。

退出条件：策略必须和 Nav2/规则基线同场对比。

第一次接管前的接口盘点

# Run on the RDK X5 after starting the relevant drivers
ros2 topic list -t
ros2 topic info -v /scan
ros2 topic info -v /image_raw
ros2 topic info -v /cmd_vel
ros2 topic hz /scan
ros2 topic bw /image_raw
ros2 topic echo --once /odom
ros2 run tf2_tools view_frames
ros2 doctor --report

如果 /odom 名称不同或不存在，以实机输出为准。不要先写一堆适配层掩盖接口没搞清楚的问题。

六、LiDAR 能否直接输入并输出动作

可以。当前雷达已经通过 ROS 2 发布 sensor_msgs/LaserScan，底盘通过 geometry_msgs/Twist 类命令控制，因此数据与执行接口都具备。真正的问题不是“能不能接”，而是怎样训练、怎样定义任务，以及怎样防止模型把车撞坏。

层级输入输出推荐算法定位
L0 规则基线scan 扇区最小距离停车/转向阈值、VFH 类方法安全基线，必须先有
L1 经典导航scan + odom + map + goalvx/vy/wzNav2 planner/controller老师策略与比较基线
L2 LiDAR-to-Action归一化 scan + goal 极坐标 + 当前速度单步或短 action chunk行为克隆、PPO、SAC、TD-MPC学习控制策略，不是 VLA
L3 多模态导航scan + RGB + goal/stateaction chunk多编码器融合、Transformer policy视觉语义加几何安全
L4 VLA 类导航语言 + RGB + scan/state目标点、技能或 action chunkVLM/VLA adapter 或分层策略语言条件机器人策略
LeRobot 可以用，但需要适配：可以给这台车实现自定义 Robot 接口，把 ROS 2 观测映射到 get_observation()，把策略动作映射到 send_action()。不过 LiDAR 不是多数现成 VLA 的标准输入，仍需自定义 feature、processor 或 policy adapter；不能因为框架能接自定义机器人，就假定预训练权重已经理解 LaserScan。

最小可行的学习策略

observation = {
  "lidar": float32[N],          # clipped, normalized, fixed angular order
  "goal":  float32[2],          # distance, bearing in base_link
  "velocity": float32[3]        # vx, vy, wz
}
action = float32[3]              # normalized vx, vy, wz proposal
  • 先把 inf、nan、越界值和雷达盲区处理规则固定；训练和实机必须完全一致。
  • scan 长度固定。不同驱动分辨率不要靠运行时静默截断，应该显式重采样并记录配置。
  • 行为克隆可用 Nav2 或人工遥控做 teacher；RL 先在仿真并行训练，再做少量真机微调。
  • 麦克纳姆动作是三维 vx/vy/wz，但第一版可以先锁住 vy=0，降低探索空间；稳定后再开放全向移动。
  • 模型输出永远是 proposal。安全监督器按距离、速度、延迟和加速度重新裁剪。
2D LiDAR 的硬限制：模型看不到扫描平面之外的物体。桌沿、楼梯、透明物、悬空物和过低障碍都可能漏检。真机学习实验只应在封闭平地、低速、有人看护且电源开关可立即触达的条件下进行；后续可补前向深度相机、ToF 或防跌落传感器。

七、4090 上的 VLA 类实验路线

不要第一天就微调一个大 VLA 让它直接发速度。最实用的路线是逐层增加不确定性，每一层都有上层基线可回退和比较。

  1. 数据契约：把图像、scan、odom/TF、目标、人工动作和模式记录成 episode；先用 rosbag2，后续再转换成 LeRobot 或自定义序列数据集。
  2. LiDAR teacher-student：Nav2 或人工遥控生成动作，训练小型 MLP/1D CNN/Transformer 模仿策略。4090 训练，先远端推理。
  3. 仿真 RL：在 Gazebo 或 Isaac Lab 中建立麦克纳姆底盘和 2D LiDAR，做碰撞惩罚、到达奖励、速度平滑奖励与域随机化。
  4. 视觉融合：加入下采样 RGB 特征，用 LiDAR 负责几何、RGB 负责语义；先预测局部目标或轨迹，再考虑直接动作。
  5. 语言分层方案：语言/图像模型识别“去蓝色椅子旁边”，输出语义目标或地图坐标，Nav2 执行。这比端到端速度更容易验证，也更安全。
  6. VLA action chunk：数据和基线成熟后，再训练“语言 + RGB + scan/state → 未来 K 步动作”。每次只执行 chunk 前几步并滚动重规划。
路线4090 可行性板端部署建议
小型 LiDAR policy很适合可继续尝试 ONNX/PTQ/BPU第一项学习控制实验
ACT/Diffusion 类模仿策略可训练先远端；是否能量化取决于算子和规模先做固定任务和短 action chunk
SmolVLA/π0/GR00T 等现成 VLA视模型与显存配置而定不适合直接放 RDK X5这些模型通常面向图像、语言和机械臂状态；LiDAR 需要自定义 adapter/processor
语言→语义目标→Nav2非常适合VLM 在 4090，导航在 RDK最先落地的“语言控制移动机器人”

八、Wi-Fi ROS 2 推理节点要点

带宽判断：1080p NV12 原始图像约为 1920×1080×1.5×30 ≈ 93 MB/s，还没算 DDS 开销，不能把它当普通 Wi-Fi topic 随便发。2D scan 通常只有几十 KB/s。第一版应发压缩、降分辨率图像，LiDAR 则可以直接发。

4090 主机最好使用原生 Ubuntu 22.04 + ROS 2 Humble，或使用能够暴露宿主网络的 Linux 容器。Windows/WSL 也可能做通，但虚拟网卡、multicast 和防火墙会额外增加 DDS 排障成本，不应作为第一版端云链路。

问题工程处理验证方式
节点互相发现两端统一 ROS_DOMAIN_ID=99、RMW 和网络接口；先测 multicastros2 multicast receive/send
Wi-Fi multicast 不稳定再引入 Fast DDS Discovery Server；不要一开始就堆网络层重复重启两端，记录发现耗时和丢失次数
图像拥塞压缩、降采样、depth 1、Best Effort；不排队处理旧帧ros2 topic bw 与端到端时间戳
时间不同步两机使用 chrony/NTP；消息保留传感器采样时间记录 source stamp、receive stamp、finish stamp
动作过期动作带 source_seq 和 valid_until；RDK 拒绝旧动作人为加入 100/300/1000 ms 延迟测试
推理节点崩溃RDK watchdog 超时输出零速度；恢复后需显式重新 armedkill 推理进程、断 Wi-Fi、拔网线
QoS 不兼容发布/订阅双方显式配置并检查ros2 topic info -v

动作链建议从 10 Hz 开始，测出 P95 延迟后再提高。控制周期不是越快越好：远端 30 Hz 但每隔几秒卡 500 ms，比稳定的 10 Hz 更危险。车端安全监督器可以 30–50 Hz 独立运行，并始终读取最新 scan。

九、BPU 部署与版本风险

RDK X5 官方规格是 10 TOPS BPU，适合检测、分类、分割和较小策略网络。官方工具链以 ONNX 模型、PTQ/量化和板端 runtime 为核心；但当前实机是 RDK OS 3.0.0，而当前 RDK Model Zoo 的主分支建议 RDK OS ≥ 3.5.0，推理接口也已演进到 hbm_runtime。

不要在现有可用系统上直接滚动升级。Yahboom 的相机、底盘、tros 工作空间和自启动程序可能依赖旧镜像。正确做法是先克隆 SD 卡，再用第二张卡验证新 RDK OS；模型转换工具链、BPU 驱动和 libdnn/hbm_runtime 必须版本匹配。
部署目标推荐路径关键验证
官方检测模型先在当前系统复现与其版本匹配的 legacy 示例模型能加载、输入 NV12 正确、后处理一致
自训视觉模型PyTorch → ONNX → 校准集 → PTQ → BPU 模型浮点/量化逐层或最终精度对比，真实场景校准集
小型 LiDAR policy先 ONNX Runtime/CPU 验证，再评估是否满足 BPU 固定 4D 输入和算子要求输入 reshape 不改变角度语义；输出与 PyTorch 对齐
大 VLA4090 推理，ROS 2 返回动作建议显存、吞吐、P95 延迟、断连行为

本机根分区实测已使用约 82%，剩余空间不适合长期存图像数据集。rosbag 应写入外接 SSD、NAS 或通过网络落到工作站，并设置按时长/大小切包和清理策略。

感知FPS、P95 延迟、温度、误检/漏检、光照条件板端持续运行 30 分钟无泄漏或降频失控
学习策略成功率、碰撞率、路径效率、平滑度、域外失败与规则和 Nav2 同场比较，不能只挑成功视频
安全断 Wi-Fi、kill 节点、旧动作、NaN、雷达掉线任何单点故障都不能让上一条非零速度持续执行

建议的前三个落地项目

  1. cmd_vel_guard：先把所有动作源统一接到 /cmd_vel_proposed，本地监督后才发布 /cmd_vel。
  2. dataset_recorder：建立可回放、可标注、带任务元数据的 episode 数据。
  3. remote_inference：先用一个非常小的 LiDAR 规则/MLP 节点打通 RDK→4090→RDK，再替换为更复杂模型。
为什么不把“直接训练 VLA”排第一

因为目前最缺的不是模型，而是稳定的数据契约、底盘标定、时间同步、网络延迟测量和本地安全监督。没有这些，训练结果差时无法判断是数据、模型、控制还是通信造成的；结果好时也无法复现。先把最简单的 LiDAR 策略跑通，随后加图像和语言只是扩展输入，不需要推翻整个执行链。
