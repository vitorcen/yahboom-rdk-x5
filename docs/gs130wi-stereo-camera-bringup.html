GS130WI 双目相机适配实录

Bring-up log: Waveshare GS130WI stereo camera (dual SC132GS global shutter) on RDK X5 — 2026-07-18
TL;DR · 小车相机由单目 IMX219 换为微雪 GS130WI 双目全局快门(双 SC132GS,1.3MP,出厂标定内置 EEPROM,板载 ICM-42688 六轴 IMU)。一天内完成:两 CSI 电气验证 → EEPROM 标定解码(基线 69.7mm,免现场标定)→ HBN 双路采集守护 → SGBM 深度 + 彩色双话题进 GUI(前端零改动)→ sensor 60fps 解锁(一个寄存器关触发模式,流中热打)→ 官方 NPU(BPU)深度栈适配(进行中)。中途实战定位并修复一起排线接触不良导致的 MIPI 物理层误码。
Swapped IMX219 for the GS130WI stereo pair; verified both CSI lanes, decoded the factory calibration EEPROM (69.7 mm baseline), built an HBN dual-capture daemon + SGBM depth / color publishers (GUI untouched), unlocked 60 fps free-run with a single register, and diagnosed a flaky FPC cable via MIPI pkt_fatal counters. 3 · EEPROM 出厂标定解码 / Factory calibration layout

1KB EEPROM,头 "UNION",数值全为小端 IEEE754 double。免现场标定,OpenCV stereoRectify 直接可用。

4 · 排线故障实战 / Flaky-cable war story

症状:左眼 ISP 后噪声 9–11(右眼 0.9)、掉帧 40%、AE 震荡;RAW 噪声却正常。
排除:settle 扫描无效、Bayer 相位一致、单路独占仍脏 → 非软件/双路冲突。
铁证:/sys/class/vps/mipi_host0/status/icnt 的 pkt_fatal 持续累积(~7/s),对侧 host 恒 0 → MIPI 物理层误码。
处置:对调板端 CSI 排线 → 全好(误码归零、满帧率)→ 归位后仍好。根因 = 排线接触不良,插拔即愈。
教训:图像"脏"先看 pkt_fatal 计数,一秒定性物理层;别在 ISP/tuning 里瞎找。

5 · 60fps 解锁 / Unlocking 60fps

真相:RDK 旧驱动表把 sensor 配成外触发模式(vts=0x3FFF 拉满),帧率完全由 X5 的 LPWM 触发脉冲决定(33333µs=30fps)。提高 LPWM 到 60Hz 无效——触发模式单帧序列 >16.6ms,每两个触发掉一个。

解法(与上游 x5-libcam-sensor 60fps master 表 diff 得出,PLL/MIPI 线速不变,流中热打、无需重编驱动):

# stream off → 时序窗口 → vts=1400 → 关触发 → stream on   (i2ctransfer w3@<addr>)
0x0100=0x00
0x3201=0x02  0x3203=0x02  0x3205=0x55  0x3207=0x15  0x3213=0x0c
0x320e=0x05  0x320f=0x78          # vts 0x0578 = 1400
0x3222=0x00                       # ← 钥匙:关外触发模式,自由跑
0x0100=0x01

已固化在 stereo_cam.py::bump_sensors_60():spawn 采集 daemon 6s 后自动对两眼热打(chip-id 探测 bus4/6,排线怎么插都行)。实测双眼 60.0fps 稳定、误码零增长。副作用:两眼失去 LPWM 硬同步,自由跑相位差每次开机随机(实测 0–9ms,固定相位,配对逻辑 60ms 容限内无碍)。

深度比彩色慢的本质:彩色是"搬运"(读+缩+转+JPEG ≈50ms),深度是"计算"(SGBM 半全局匹配 ≈260ms@432×512×64 视差)——差两个数量级,是算法成本不是管线 bug。

6.4 · 阶段B:C++ 热路径 + 源头配对 / Stage B: C++ hot path & pair-at-source (5.4→16.8fps)

  • 三层瓶颈逐个击破:① python 单进程 color+combine 双线程共抢 GIL,合计上限 ~11 it/s;② 消费端轮询配对每次全量读 2MB×2 烧 tmpfs 带宽(144% CPU 只换来 6fps);③ 最致命——系统负载下两眼各自独立丢帧 ~13%,"半帧内近对必存在"的前提崩塌(实测配对 dt p50=12ms > 8.5ms 门限)。
  • 正确的数据结构:配对下沉到 stereo_capture(两眼数据都在它手里)。每眼 staging 最新帧,后到者查对方时间戳,±8.5ms 内即 seqlock(序号奇偶锁)写单一 /dev/shm/stereo_pair.shm(64B 头 + 双 NV12,mmap 零 syscall,PAIR_HZ 默认 30)。消费端 mmap 快照,零轮询、零配对逻辑。
  • C++ 节点 stereo_combine_pkg 接管 remap(fixed-point CV_16SC2 直写 msg 缓冲)+ 彩色(缩放/径向 chroma/JPEG);stereo_cam.py 退居监督者(EEPROM 标定 → 导出 rect_maps.bin,拉起看护 capture/stereonet/codec/combine 四进程;COMBINE_BACKEND=py 回退)。
  • infer_thread_num:=4(默认 2)加深推理流水线:BPU 24%→63%。实测(唯一载荷 md5 计数):深度 16.8fps、彩色 10.1fps,进入官方 15-27 区间;当前瓶颈是 stereonet 推理本身(27fps 为无渲染裸推理口径)。
  • 测量陷阱:ros2 topic hz 订阅 1.35MB 大图会因 python 反序列化跟不上而漏计;stereonet 自绘 FPS 角标口径不可信——以唯一 JPEG 载荷计数为准。

6.5 · 彩色偏粉的真相 / Why color is never "full color": no IR-cut filter

  • 取证:抓 pre-ISP RAW(STEREO_TAP=vin)做 2×2 相位统计——绿对角相等(64.7/64.7)、蓝位低(48.8)、红位高(60.7),确认是彩色 BGGR Bayer 传感器,不是黑白。
  • 根因:模组无 IR-cut 滤光片(规格卖点即 850/940nm NIR 增强)。室内灯的近红外穿透 R/B 滤色层 → 整片品红加性污染,且带镜头径向 shading。这是传感器端光谱污染,任何 CCM/WB 矩阵都不可逆。
  • 3A 侧症状:logcat 里 AWB 永远 "can not calculate right color temperature"(白点在自然光轨迹外)+ 无 OTP;色温=0 → CCM 不套用。calib_lname tuning json 实际已加载,不是没加载。
  • 已落地的软件上限:① stereo_capture 启动 3s 后冻结手动 WB(freeze_isp_wb,默认锁 auto 当前值,可 ISP_WB_GAINS="r,b" 覆写,ISP_WB_TEMPER=0 恢复 auto),消除 AWB 漂移;② stereo_cam.py 的 chroma_fix:半分辨率 UV 平面按径向环带扣除 NIR 色度基底 + 1.6× 饱和。局部真实色(红物件/绿凳)存活,大面积表面被中和成米灰(均值扣除的数学必然)。
  • 真修复只有硬件:加 650nm 短通 IR-cut 滤光片(牺牲夜视),或换带切换式 IR-cut 的模组。当前视觉效果对"态势感知窗口"够用;要纯净图可发 Y 通道灰度。
