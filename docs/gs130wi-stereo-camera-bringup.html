TL;DR · 小车相机由单目 IMX219 换为微雪 GS130WI 双目全局快门(双 SC132GS,1.3MP,出厂标定内置 EEPROM,板载 ICM-42688 六轴 IMU)。一天内完成:两 CSI 电气验证 → EEPROM 标定解码(基线 69.7mm,免现场标定)→ HBN 双路采集守护 → SGBM 深度 + 彩色双话题进 GUI(前端零改动)→ sensor 60fps 解锁(一个寄存器关触发模式,流中热打)→ 官方 NPU(BPU)深度栈适配(进行中)。中途实战定位并修复一起排线接触不良导致的 MIPI 物理层误码。

3 · EEPROM 出厂标定解码 / Factory calibration layout

1KB EEPROM,头 "UNION",数值全为小端 IEEE754 double。免现场标定,OpenCV stereoRectify 直接可用。

偏移字段实测值
0x00魔数 "UNION" + 版本字节—
0x18cam1(↔0x30 眼):fx fy cx cy + k1 k2 p1 p2(+4 零)666.49 / 666.68 / 526.79 / 637.45
0x78cam1 重投影误差0.226 px
0x81cam2(↔0x32 眼)同布局658.36 / 658.59 / 520.23 / 648.67

cam1↔0x30 的对应关系用 SGBM 有效率实证:正确对应 41.5% vs 交换 23.1%。标定按 1088×1280 全分辨率,降尺度使用时 K 同比缩放。

4 · 排线故障实战 / Flaky-cable war story

症状:左眼 ISP 后噪声 9–11(右眼 0.9)、掉帧 40%、AE 震荡;RAW 噪声却正常。
排除:settle 扫描无效、Bayer 相位一致、单路独占仍脏 → 非软件/双路冲突。
铁证:/sys/class/vps/mipi_host0/status/icnt 的 pkt_fatal 持续累积(~7/s),对侧 host 恒 0 → MIPI 物理层误码。
处置:对调板端 CSI 排线 → 全好(误码归零、满帧率)→ 归位后仍好。根因 = 排线接触不良,插拔即愈。
教训:图像"脏"先看 pkt_fatal 计数,一秒定性物理层;别在 ISP/tuning 里瞎找。

5 · 60fps 解锁 / Unlocking 60fps

真相:RDK 旧驱动表把 sensor 配成外触发模式(vts=0x3FFF 拉满),帧率完全由 X5 的 LPWM 触发脉冲决定(33333µs=30fps)。提高 LPWM 到 60Hz 无效——触发模式单帧序列 >16.6ms,每两个触发掉一个。

解法(与上游 x5-libcam-sensor 60fps master 表 diff 得出,PLL/MIPI 线速不变,流中热打、无需重编驱动):

# stream off → 时序窗口 → vts=1400 → 关触发 → stream on   (i2ctransfer w3@<addr>)
0x0100=0x00
0x3201=0x02  0x3203=0x02  0x3205=0x55  0x3207=0x15  0x3213=0x0c
0x320e=0x05  0x320f=0x78          # vts 0x0578 = 1400
0x3222=0x00                       # ← 钥匙:关外触发模式,自由跑
0x0100=0x01

已固化在 stereo_cam.py::bump_sensors_60():spawn 采集 daemon 6s 后自动对两眼热打(chip-id 探测 bus4/6,排线怎么插都行)。实测双眼 60.0fps 稳定、误码零增长。副作用:两眼失去 LPWM 硬同步,自由跑相位差每次开机随机(实测 0–9ms,固定相位,配对逻辑 60ms 容限内无碍)。

深度比彩色慢的本质:彩色是"搬运"(读+缩+转+JPEG ≈50ms),深度是"计算"(SGBM 半全局匹配 ≈260ms@432×512×64 视差)——差两个数量级,是算法成本不是管线 bug。

6 · NPU(BPU)深度加速 / Official BPU StereoNet stack

RDK 官方对这款相机的深度方案就是跑 10TOPS BPU:

mipi_cam 双目模式(认 SC132GS,出 /image_combine_raw 左右拼合图)
  → hobot_stereonet(BPU 神经网络 DStereoV2.x,int8/int16 × 多档分辨率,标称 15-25fps)
  → hobot_codec(硬件 JPEG)→ websocket 预览
