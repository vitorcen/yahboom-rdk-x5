Dataset Recorder 设计Gamepad one-key episode recording & playback

RDK X5 Robot · rosbag2 · 64G 卡已扩容(余 ~60%) · v2 2026-07-13(gpt-5.6 评审后修订) · 已实施,板上实录验证
TL;DR — 板端新增常驻节点 episode_recorder.py,复用 safety_stop 的 “节点持有状态 + 外部只发翻转 + latched 广播镜像”单一机制:手柄 START 键与 GUI ⏺ 按钮 都只发 /record_toggle(Empty),全停链路另走幂等 /record_stop (评审修订:全停发 toggle 会在未录制时反向开录);节点起停 ros2 bag record 子进程、 写 meta.yaml、蜂鸣反馈,latched /recording(Bool) 供两端镜像显示。 分析/回放放在工作站新 notebook episode_lab.ipynb(列出/拉取/画轨迹/回看图像); strafe_test.ipynb 原样保留,它是横移硬件故障的验收工具,不混用。 One resident recorder node on the board, toggled by gamepad START or GUI; analysis lives in a new workstation notebook; the strafe diagnosis notebook stays untouched.

一、为什么是这个形态Why this shape

  • 录制必须离得开电脑 — 人在场地里拿手柄示教,Jupyter 开没开不该有关系。 录制器是板端 systemd 常驻节点,断 Wi-Fi 照录。
  • 单一机制,零特殊情况 — 急刹开关(/safety_toggle + latched /safety_enabled)已经验证过这套模式:状态只有一份、住在节点里, 手柄和 GUI 都是“发翻转 + 镜像显示”,不存在两端状态打架。录制照抄。
  • notebook 只做事后分析 — 交互画图、逐帧回看、标注成败,这才是 notebook 的正确用途。 新开 episode_lab.ipynb,不改造 strafe_test.ipynb (后者是未解决的后轮扭矩故障的验收工具,换轮后还要用)。
  • 这是后续一切学习实验的地基 — LiDAR-to-Action 行为克隆、4090 远端推理对比、 Nav2 基线统计,全都吃同一种 episode 数据。先固定数据契约,模型随便换。

三、板端录制节点episode_recorder.py

3.1 接口契约 / Interface

话题类型方向语义
/record_togglestd_msgs/Empty入翻转录制状态(手柄 START / GUI 按钮都发这个)
/record_stopstd_msgs/Empty入幂等停止,全停链路专用——评审抓出的 bug:全停若发 toggle,未录制时会反向开录
/recordingstd_msgs/Bool,latched出当前是否在录,开机默认 False;GUI 亮红点、手柄无屏靠蜂鸣
/Buzzerstd_msgs/Bool出声音反馈,见 3.4 约定表

3.2 录制内容 / What gets recorded

ros2 bag record -o ~/episodes/ep_<YYYYmmdd_HHMMSS> \
  /scan /odom /tf /tf_static \
  /cmd_vel /cmd_vel_joy /cmd_vel_follow /cmd_vel_mux /cmd_vel_drv \
  /joy /safety_enabled /image_jpeg \
  --max-bag-size 1073741824 \       # 1 GiB 切包(512MB 高负载切包有丢消息风险,rosbag2#2108)
  --qos-profile-overrides-path qos_overrides.yaml   # /tf_static /safety_enabled
                                    # 显式 transient_local,否则 latched 历史录不进
  • 录 全部四级 cmd_vel:这样每个 episode 自带“谁在开车、mux 裁了什么、 safety_stop 压了多少速”的完整证据链,事后不用猜。
  • /image_jpeg 是压缩流(GUI 同款),按 640×480@~15Hz 估约 0.5–1 MB/s,10 分钟 episode ≈ 0.3–0.6 GB,64G 卡余 60% 完全够; 参数 record_image(默认 true)可关。
  • 雷达+odom+tf+cmd_vel 合计只有几十 KB/s,忽略不计。

3.3 元数据 / meta.yaml

停止录制时由节点写入 episode 目录,张量不带上下文将来一定解释不了(路线图文档的原话):

# ~/episodes/ep_20260713_153012/meta.yaml
episode: ep_20260713_210340
start: 2026-07-13T21:03:40+08:00
duration_s: 42.3
missing_topics: []                  # 开录时不在图上的话题,不静默缺列
stopped_by: toggle | stop_all | disk_full | shutdown
task: null                          # 事后标注写本地 annotations.yaml,
success: null                       # meta.yaml 本身不回改(评审建议:
notes: null                         # 原始记录不可变,标注另放一层)

录制中目录带 .partial 后缀,子进程退出、meta 落盘后才 rename 成正式名—— 残包一眼可辨,不承诺"半截 bag 一定可读"(评审建议)。

3.4 蜂鸣约定 / Beep conventions

已占用:急刹开=两短,急刹关/全停=一长。录制取三短系,不与既有冲突:

事件声音备注
开始录制三短滴与“急刹开=两短”区分
停止录制(正常落盘)一短一长“收尾完成”
拒绝开始(磁盘不足)五连急促短滴见 3.5 守门

3.5 磁盘守门与安全 / Guards

  • 余量守门:开始前查 ~/episodes 所在分区,余量 < min_free_gb(默认 4 GB)则拒绝并急促蜂鸣。录制中每 30 s 复查, 跌破即自动停止落盘(宁可断录不可写满根分区)。
  • 优雅收尾:停止 = 给子进程发 SIGINT 并等待退出, rosbag2 自己写完 metadata;SIGKILL 只在超时(5 s)后兜底。
  • 与全停联动:手柄 Y / GUI 🛑全停触发时,若在录,顺带停止录制并在 meta 里记 stopped_by: stop_all —— 全停通常意味着这条 episode 出事了,这本身就是要的标注。
  • 崩溃自愈:节点由 nav-bringup respawn=True 拉起;重启后状态回到 “未录制”(半截 bag 已由 rosbag2 尽力落盘,notebook 端能识别缺 metadata 的残包)。

五、工作站分析 notebookepisode_lab.ipynb

新文件,与 strafe_test.ipynb 并列;沿用其“ssh 灌板上执行”的成熟套路。cell 规划:

  1. 列出 — ssh 列 ~/episodes:名称、时长、大小、task/success 标注状态,表格输出。
  2. 拉取 — rsync 指定 episode 到本地 episodes/(git-ignore)。
  3. 导出 — 板上没有的依赖不装:用一段灌到板上的 rosbag2_py 脚本把 odom/cmd_vel/scan 摘要导成 CSV 随包拉回(工作站无 ROS 也能分析;JPEG 帧本来就是标准 JPEG, sqlite3 直读即可回看)。
  4. 可视化 — odom 轨迹俯视图(叠加地图可选)、四级 cmd_vel 时间线对比 (直观看到 mux 仲裁与 safety_stop 压速)、scan 抽帧极坐标图。
  5. 回看 — 图像帧滑条逐帧浏览,对照时间戳。
  6. 标注 — 填 task/success/notes 写本地 annotations.yaml(独立文件,meta.yaml 与原始 bag 不可变); 只留工作站副本,板上是采集缓存,单向流动不搞双向同步。
  7. 演示重放(可选) — ssh 触发板上 ros2 bag play --topics /cmd_vel。 仅演示用:走 mux 最低优先级,初始位姿不同轨迹必漂, 不能当验证手段;真正的“回放验证”是把录的观测喂给策略离线比对,那是训练侧的事。

六、验收清单Acceptance checklist

  1. 手柄 START 开录 → 三短滴;GUI 红点同步亮;开 1 分钟遥控 → START 停 → 一短一长, meta.yaml 完整。Start/stop from gamepad, GUI mirrors.
  2. GUI 按钮开录、手柄停录(交叉操作),状态不打架。
  3. 录制中按 Y 全停:车停 + 录制自动收尾 + stopped_by: stop_all。
  4. 录制中 kill -9 recorder:respawn 拉起、状态归 False、残包可被 notebook 识别。
  5. 用 fallocate 占满磁盘至阈值下:开录被拒 + 急促蜂鸣;录制中跌破 → 自动停。
  6. notebook:列出/拉取/画轨迹/图像回看全通;四级 cmd_vel 时间线能看出 safety_stop 压速痕迹。
  7. 断 Wi-Fi 全程录制不受影响(录制器不依赖工作站)。

七、明确不做Non-goals

  • 不改造 strafe_test.ipynb —— 它是横移硬件故障(后轮扭矩弱,未解决)的验收工具,保持原样。
  • 不做板上 Web 标注界面 —— 标注在 notebook 里做,单向数据流。
  • 不做 LeRobot 格式转换 —— 先 rosbag2 落盘;将来训练需要时加一个离线转换脚本即可, 数据契约(话题集合+meta 字段)本设计已固定。
  • 不做双向 meta 同步 —— 工作站副本为准。
实施顺序 — ① 上板抓 START 键编号 → ② episode_recorder.py + launch 接线 → ③ joy_teleop.py / GUI 各加一个翻转入口 → ④ 板上实录三条 episode 过验收 1–5 → ⑤ episode_lab.ipynb 过验收 6。全程不碰 mux/safety_stop 的控制链路,零破坏。
