为什么原脚本一直提示“扫描不到 SSID”

目标 SSID 实际存在，电脑能够看到并连接。失败来自板端网络栈的三处冲突，而不是路由器信号或密码。

问题现场表现修复
wpa_supplicant.service 被 mask NetworkManager 报 Failed to D-Bus activate wpa_supplicant，wlan0 为 unavailable client 切换前解除 mask 并启动服务
厂商 XFCE 自启动强制恢复 AP Open_AP.desktop 延迟执行 open_AP.sh，重新停止 supplicant、启动 hostapd 进入 client 前将该自启动项设为 Hidden=true
AIC8800 的 5 GHz 国家码限制 默认管制域可能只能看到 2.4 GHz，目标信道 44 不出现 配置 country_code=CN custregd=N 并在需要时重载驱动
关键判断：“扫描列表为空”并不一定代表周围没有 Wi-Fi。本次首先是 supplicant 不可用，其次还有厂商 AP 自启动竞争；只反复执行 nmcli device wifi rescan 无法解决。

修正版脚本的切换流程

01
建立回滚保护 注册独立的 90 秒 systemd 回滚任务，主脚本卡住也能恢复热点。
02
释放 AP 停止 hostapd 与 DHCP Server，清理 wlan0 的 AP 地址。
03
修正无线驱动 设置 AIC8800 国家码，确保可扫描 5 GHz 信道 44。
04
启用客户端栈 启动 wpa_supplicant，将 wlan0 切成 managed 并交给 NetworkManager。
05
连接并持久化 连接目标 SSID、获取 DHCP 地址，禁用厂商 AP 自启动并取消回滚任务。

安全保护

  • nmcli 扫描和连接均设置超时，避免驱动调用长期卡死。
  • 如果找不到目标、supplicant 启动失败或连接失败，脚本立即调用 wifi_ap.sh。
  • 即使主脚本卡死，独立的 90 秒回滚任务仍会尝试恢复 RDK_X5_Robot。
  • NetworkManager 已经自动连接目标时，脚本直接识别当前 SSID，避免无意义的重复扫描。

日常使用

脚本部署在板端 /home/sunrise/scripts/，仓库的 scripts/ 目录保留相同版本。

连接默认路由器

sudo /home/sunrise/scripts/wifi_client.sh

脚本会在终端静默提示输入家庭 Wi-Fi 密码；密码不会写入项目文件。

临时指定其他 Wi-Fi

sudo /home/sunrise/scripts/wifi_client.sh "SSID" "PASSWORD"

命令行参数可能被 shell 历史记录。需要避免明文历史时，直接省略密码并使用交互提示。

同时指定国家码

sudo /home/sunrise/scripts/wifi_client.sh "SSID" "PASSWORD" CN
SSH 会断开：如果电脑当前连接的是板载热点，client 切换成功后 192.168.8.88 会立即失效。这是预期行为。电脑需改连目标路由器，再使用板子的新 DHCP 地址登录。

为什么 DHCP Server 显示 failed 不影响联网

isc-dhcp-server 是板子作为 AP 时使用的服务：它给连接到 RDK_X5_Robot 的电脑或手机分配 192.168.8.x 地址。

模式谁给谁分配地址板端 isc-dhcp-server
Client 家庭路由器给 RDK X5 分配地址 不需要，inactive/failed 不影响板子联网、SSH、DNS 或 ROS
AP RDK X5 给接入热点的设备分配地址 必须正常，否则设备可能连上热点却拿不到 IP
client 模式真正需要关注的是 wpa_supplicant、NetworkManager、wlan0 地址和默认路由，而不是 AP 专用的 DHCP Server。

恢复板载 AP 热点

sudo /home/sunrise/scripts/wifi_ap.sh

恢复成功后：

  • 热点 SSID：RDK_X5_Robot
  • 热点密码：12345678
  • 板端地址：192.168.8.88
  • SSH：ssh root@192.168.8.88

wifi_ap.sh 会重新 mask wpa_supplicant、启动 hostapd 和 DHCP Server，并恢复厂商 Open_AP.desktop 自启动项。因此执行 AP 恢复后，后续重启仍会保持 AP 模式。
