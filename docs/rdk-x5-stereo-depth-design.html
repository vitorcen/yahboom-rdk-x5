一、硬件核验门Gate 0: restore a known baseline first

已核实的板载事实(2026-07-15 板勘)

  • 板型 board_id=302(RDK X5 V1.0),/etc/board_config.json 声明 camera_num=2:host0 ↔ i2c-6(板上丝印 CSI1)、host2 ↔ i2c-4(板上丝印 CSI2)。 SoC 侧只有两路相机 VIN 资源可用 —— 这是硬件事实,不是配置项。
  • GS130W = 双 SC132GS(1280×1080@120fps,157° DFoV,基线 ≈0.08 m,出厂标定在模组 EEPROM), 两根 22pin 排线占满 host0+host2。排线有 CAM/RDK 方向标注,插反=全灭。
  • 板上已有 libsc132gs.so(/usr/hobot/lib/sensor/),但 3.0.0 内核/VIN 侧 对 SC132GS 双目的支持待验(论坛有旧镜像缺驱动先例)。
  • 第三个 22pin 座 = DSI 显示口(已确认,弃用):X5 官方硬件=2×CSI + 1×DSI,外观相同, 之前 IMX219 接到这里是接错口 —— 插 DSI 永远不会被认到,正是当前 i2c 全空的原因之一。 本方案不再考虑三口并存,IMX219 退役,两个 CSI 全给双目。

核验顺序(严格串行,每步是下一步的对照组)

  1. 断电核对:两个 CSI 座丝印/排线方向/锁扣;DSI 座保持空置(IMX219 已从错口拔下)。
  2. 恢复已知基线:用确认好过的 IMX219,在 host0、host2 分别单独点亮 (i2cdetect 出 0x10 + mipi_cam 出图)。这一步证明两个座、cam-service、 /dev/isc、3.0.0 采集链都活着 —— 排除"插拔弄坏了什么"。
  3. GS130W 逐目探测:左右目排线分别单插,i2cdetect -y -r 4 / -r 6 期望 0x32/0x33(+0x50 EEPROM);若扫出 SC230AI 地址段则是另一款模组,路线重定。记录每一步失败点。
  4. 双目合采:两线齐插,dual launch 上电,验 LPWM 硬同步与左右顺序(遮镜法:蒙左镜头看哪路黑)。
# 板上(ROS_DOMAIN_ID=99):
i2cdetect -y -r 4 ; i2cdetect -y -r 6
dmesg | grep -iE 'sc132|imx219|csi|vin'
systemctl stop mipi-cam        # 试运行前停旧服务,避免抢 host
基线结论(已定):两口全给双目,IMX219 退役。 第三座是 DSI 显示口不是相机口,"IMX219 迁 CSI 并存"物理不成立。§四保留最小配置能力,只为 IMX219 将来换 USB 摄像头或走"双目左目回喂"时能接回,不是为了现在并存。

二、数据通路dataflow: dual MIPI → BPU → JPEG → rosbridge → GUI

关键设计决定(含 codex 修正)

  • 不自研立体匹配:hobot_stereonet 官方 BPU 方案,X5 15–25 fps。唯一自建=一个极薄 launch (no-web StereoNet + 独立 codec 实例)。
  • 不启官方 web-visual launch:它自带 websocket 抢 :8000,与现有单目预览端口冲突; 且 stereonet_pub_web=false 会连 codec 一起关。所以用 no-web launch + 自己的 hobot_codec(codec_in_mode=ros, in_format=bgr8)发 /stereonet_visual_jpeg。GUI 走 rosbridge,根本不需要第二个官方网页。
  • 伪彩在板上渲染,GUI 只画 JPEG:深度原图走 rosbridge base64 带宽自杀; stereonet_visual(bgr8 伪彩,官方链路本就喂 codec)压 JPEG 后与现有相机预览同构。
  • 板内话题裁剪:origin_left/right、右目校正图、pointcloud 全关 (640×352 NV12 双目 30fps 板内裸流 ≈20MB/s,先爆的是内存带宽/DDS 拷贝,不是 WiFi)。 只留 combine、depth、visual。队列深度 1,sensor-data/best-effort,允许丢旧帧。
  • 限流位置明确:在 codec 前(StereoNet 可用 infer_thread_num=1 压推理频率, 或加薄节流节点)。GUI 少画几帧不减板端成本,只算渲染保护。
  • QoS 逐边验收:TROS 图像链是 best-effort;每条边用 ros2 topic info -v 核对 publisher/subscriber 匹配,rosbridge 实收帧率单独测,不以 topic list 为准。
  • hbmem 边界:Hobot hbmem 零拷贝消息 rosbridge 序列化不了。板内高带宽链可用 hbmem; 凡暴露给 rosbridge/recorder/现有节点的出口,必须是普通 sensor_msgs/Image|CompressedImage。
  • 右窗全彩图 = GS130W 左目(第一阶段交付,用官方原生左目话题):SC132GS 是彩色 sensor。 不自研 splitter —— 官方 hobot_mipi_cam 用 dual_combine=1 本就直接发布 /image_left_raw//image_right_raw(除 /image_combine_raw 外)。 直接把 /image_left_raw 接独立 codec 发 /image_jpeg(名字不变 → 右窗/录制/:8000 契约保留), 全分辨率、独立于 StereoNet 推理帧率。先做实测门:升级后 ros2 topic list 确认原生左目话题 存在及其 encoding/分辨率/帧率,存在即用、关掉不需要的右目独立输出。 只有原生左目确实不可用时才退到:①官方 /StereoNetNode/rectify_left_image(640×352、帧率绑推理); ②自研 NV12 左半 splitter(并不"薄":按实际 width/height/step 逐行复制 Y/UV 双平面,处理 1080/1088 padding/stride/旋转/左右序,禁止 data 对半切,必须色块+遮镜+奇偶行+连续帧校验,防"能编码但颜色/左右悄悄错")。
  • follow-me 的 BPU 几何 = 独立攻关(不阻塞预览):彩色预览窗好办,但 follow-me 的 人体/手势检测 + follow_me.py 追踪把 /image_raw+960×544+HFOV 62°+ROI/方位角/增益 写死在 IMX219 几何上;换成 157° 超广角左目后这些全要重标定。攻关清单:检测节点对左目尺寸的 接受度 → follow_me.py 几何参数化(IMG_W/H/HFOV 进配置)→ 手势/跟随实测。 通过之前跟随功能置灰并标注原因,但右窗彩色预览照常。

四、相机模式切换mode switch, reusing the proven mapping↔nav pattern

需求是切换:现在双目(左深度+右全彩),以后可能切回单相机。两目占满两口、单相机只用一口, 物理上互斥 —— 这与项目里 mapping.service ↔ nav2 用 Conflicts= 整栈互换是同一个问题。不发明新机制,直接复用这套(板上还有现成 camera_mode.sh 可扩展)。

# /home/sunrise/nav_config/camera.conf   (board/ 镜像同步)
# Current mode. camera_mode.sh writes this; services read it; GUI mirrors it.
CAMERA_MODE=stereo            # stereo | mono
# Single-cam params (only used in mono mode). channel = mipi host, legal 0|2.
MONO_SENSOR=IMX219
MONO_CHANNEL=0
  • 互斥靠 Conflicts=,但不是零风险照搬:Conflicts= 只保证冲突、不保证先停净再启。 两 unit 都要补:After= 建立 stop-before-start 顺序、KillMode=control-group + TimeoutStopSec= 确保多进程 launch 的全部子进程(mipi/stereonet/codec)被干净收掉、 验收"停止后无残留进程、无相机设备占用"(否则新栈在旧 VIN/codec 未释放时抢起来)。 camera_mode.sh stereo|mono = flock 防并发 + 原子写 conf + systemctl start 目标 + 失败回滚。开机只 enable 一个 unit(按 conf 拉起,两个都 enable 会启动事务竞争)。 骨架是 mapping.service Conflicts=nav2 的翻版,但这些边界条件要逐条落。
  • 两模式共用 /image_jpeg 契约:双目模式由左目 codec 发,单目模式由单相机 codec 发 —— 右窗/录制/:8000 永远订同一个话题,切模式对它们透明(Never break userspace)。
  • 深度只在双目模式存在:/stereonet_visual_jpeg 单目模式不发,GUI 左窗自动隐藏。 不需要 GUI 显式判模式 —— 这是初稿 RGB_SOURCE 抽象被 codex 否掉后的正解: 用"话题有没有流"表达模式,而不是塞一个前端状态变量。
  • GUI 不包装物理热插拔:官方明确禁止带电插拔相机,所以"换单相机"= 关机 → 换线 → 开机按 conf 启动, 不是点一下 GUI 就切。GUI 侧只做两件事:①显示当前模式(读 camera_mode 的 invoke,像 mFollow 那样反映真实状态);②双目↔单目纯软件重启(排线已到位、不涉插拔时)可给一个 invoke('camera_mode',{mode}) 按钮触发 camera_mode.sh,但明确"切相机需先断电换线"。 日常最省事路径还是改 conf + 重启,不追求 GUI 一键。
  • GUI 渲染无模式状态,但控制有:右窗/左窗靠"有流才显示 + 掉流看门狗"自动显隐(已实现), 不需前端判模式;但模式按钮的当前态、切换中/失败提示仍是显式状态。切换存在停旧启新的断流窗口, §七 G4 的"不断流"应理解为"断流限制在数秒内、不混流",看门狗保证旧帧不残留。

五、GUI 深度窗last step, not first

  • 感知链验收(§七 G1–G3)全过之后才动 GUI —— 浮窗抽取没有技术风险价值,不许阻塞主线。
  • 左上角 #depthbox(top:10px; left:10px),与右上 #cambox 同款拖拽/等比拉伸 —— 把 camera.js 的拖拽逻辑抽成 makeFloatBox() 公用,随本步一起做。
  • 订阅 /stereonet_visual_jpeg → base64 img,有流才显示,>2s 无帧隐藏; 仪表条加 iDepth 新鲜度灯;窗角一行伪彩图例(红近蓝远),不做测量标尺。
  • 带宽预算:640×352 JPEG ≈30–60KB/帧 ×10fps ×1.33(base64) ≈0.5–0.8MB/s,叠加相机流后弱网角落实测 rosbridge 实收帧率与端到端延迟(不是只看 topic hz)。

  4. G3 ROS 双出口:独立 codec 发 /stereonet_visual_jpeg(深度)+ 官方 /image_left_raw 接 codec 发 /image_jpeg(全彩,先实测原生左目话题存在);普通订阅者与 rosbridge 均实收 (QoS 逐边核对),板端 CPU/BPU/内存记录在案。

  6. G5 GUI:✅ 已提前实现(纯前端,对既定话题契约,浏览器模式可结构验证): 左窗深度 #depthbox + 右窗全彩 #cambox + makeFloatBox() 抽取共用 + iDepth 新鲜度灯 + 有流才显示的自动显隐。待板上出话题后验弱网实收帧率/延迟。
  7. G6 回归与收尾:遛狗/导航/录制回归(跟随按几何停用状态验证提示正确);系统 Tab 板卡图更新 (CSI 两路→双目);systemd 自启;follow-me 左目几何攻关另行立项。

八、三问自检the three questions

  • 真问题? 是 —— 深度感知补 2D 雷达的高度盲区(桌面/悬空物),是数据集与 VLA 的传感基础。
  • 更简单方案? 主线已最简:官方组件 + 一个薄 launch + 一个 service + GUI 一个窗。 被砍掉的复杂化:自研立体匹配、GUI 传原始深度自己上色、RGB_SOURCE 运行时切换抽象、GUI 选口设置页、 第二个官方网页预览。
  • 破坏什么? 右窗彩色/录制/:8000 靠 /image_jpeg 契约保留,切模式对它们透明; 唯一诚实停用的是 follow-me 的跟随/手势功能(几何依赖 IMX219,换超广角左目要重标定,独立攻关门)。 系统级风险(apt 连带/BPU 争抢)各有判定标准与回滚手段。
